# GEA EOR SaaS 上线前测试与验收方案

## 1. 验收策略概览

针对即将上线的 GEA EOR SaaS Admin 系统，本方案旨在提供一套全面的上线前测试与验收指南。考虑到系统具有小基数用户群和极高准确度要求（尤其是财务、薪资及合规模块）的特点，验收策略将侧重于人工流程验收、核心业务闭环验证以及针对关键风险点的自动化保障机制。

本方案分别从产品经理、开发工程师和测试工程师三个专业视角进行深入剖析。产品经理视角着重于业务流程的完整性、多门户体验的连贯性以及数据流转的准确性。开发工程师视角聚焦于系统架构的并发安全、定时任务（Cron Jobs）的健壮性以及部署与灾备策略。测试工程师视角则提供详细的边界测试用例、权限隔离验证以及自动化测试的补充建议。

---

## 2. 产品经理视角：业务闭环与体验验收

产品经理在验收过程中的核心职责是确保系统业务逻辑形成完整闭环，并且三大门户（Admin 后台、Client Portal 客户门户、Worker Portal 员工门户）之间的数据流转准确无误。

### 2.1 核心业务生命周期验证

员工全生命周期管理是 EOR 业务的核心。验收过程必须覆盖从入职、在职异动到离职的完整流程。在员工入职阶段，需要在 Admin 后台创建草稿状态的员工，并推进至合同签署状态，随后验证当系统日期到达预定入职日期时，系统能否自动将其状态更新为激活状态。特别需要注意的是，对于在当月 15 号之后入职的员工，系统应自动生成按工作日比例计算的 Sign-on Bonus，并确保该笔奖金不会被错误地并入当月薪资单，而是正确顺延至次月处理。

在员工在职期间，休假与异动模块的验收同样关键。测试人员需分别登录 Client Portal 和 Worker Portal，提交各类休假申请及费用报销。随后，必须验证审批流能否按照“提交-客户审批-管理员审批”的既定路线顺畅流转。此外，跨月休假申请必须被系统自动拆分为对应月份的两条独立记录，以保证薪资扣减的准确性。

员工离职流程的验收则涉及状态的最终流转与财务结算。当发起离职流程后，员工状态应转为离职中，并在到达最后工作日时自动更新为已终止。此时，系统必须自动触发押金退还逻辑，生成相应的信用票据（Credit Note）并将其金额转入客户的冻结钱包中。更为重要的是，必须验证薪资定时任务的“遗弃数据兜底（Orphan Rescue）”机制，确保离职员工在离职后补交的报销或异动数据能够被系统成功抓取，并为其生成最终的结算薪资单。

### 2.2 账单发票闭环与门户体验

账单与发票的生成是系统财务合规的最后一道防线。验收时需确认系统能否根据薪资单、管理费及各项税费准确生成客户发票。发票的状态流转必须经过草稿、待审核、已发送至已支付的完整路径。在发票从草稿转为待审核状态时，系统应自动从客户钱包中扣除相应金额。若发票被驳回至草稿状态，系统必须能够准确执行钱包扣款的回滚操作，恢复客户余额。

在多门户体验方面，Client Portal 必须严格执行基于客户 ID 的数据隔离，确保客户仅能查看本公司的员工、发票与报表数据。同时，休假申请页面的员工选择器必须准确包含激活、休假中以及离职中状态的员工。Worker Portal 则需保证员工登录后仅能访问个人的工资单、休假余额及报销入口。此外，系统的中英双语切换功能必须在邮件模板、发票 PDF 以及所有前端界面中稳定生效，且所有对外发出的邮件均需正确展示 GEA 品牌标识。

## 3. 开发工程师视角：架构安全与健壮性验收

开发工程师的验收重点在于确保系统在生产环境下的并发安全性、定时任务的准确调度以及部署架构的稳定性。

### 3.1 定时任务健壮性审查

系统依赖一系列定时任务来驱动业务状态的自动流转。经过代码层面的核查，早期审计报告中指出的薪资聚合问题已被修复。在上线前，必须验证这些任务的执行时间是否已合理错峰。例如，员工激活与终止任务安排在每月 1 号的 00:01，自动锁定任务在 5 号的 00:00，而薪资单生成任务则在 5 号的 00:05 执行。这种时间安排可有效避免数据库读写冲突。此外，必须验证服务器时区配置是否已强制设定为北京时间（UTC+8），以确保所有定时任务的触发时间与业务预期完全一致。

定时任务的幂等性也是审查的重点。无论 `runAutoCreatePayrollRuns` 任务因何种原因被重复触发，系统都必须保证不会为同一员工在同一周期内生成重复的薪资单。同时，针对离职员工遗留数据的兜底抓取机制，必须在预发环境中构造极端测试数据，以验证该机制的实际有效性。

### 3.2 数据库并发安全与系统配置

钱包服务（Wallet Service）是系统中最敏感的财务模块。该模块目前采用了基于版本号（version）的乐观锁机制来控制并发更新。上线前，开发团队需要编写并发测试脚本，模拟多个请求同时对同一钱包进行充值或扣款操作，以验证系统能否正确抛出并发冲突错误，而非默默覆盖数据。同时，发票状态变更与钱包资金变动必须被包裹在同一个数据库事务中，确保在任何异常情况下都能实现数据的完整回滚。

在安全与合规性配置方面，必须确保生产环境配置文件中的 JWT 密钥已替换为高强度的随机字符串，且未被提交至版本控制系统中。鉴于系统采用了三个独立的子域名（Admin, App, Worker），当前的 Cookie 配置采用了 `sameSite: "none"` 与 `secure: true` 的组合，开发人员需在真实浏览器环境中彻底验证跨域登录状态的稳定性。此外，系统已配置的 API 速率限制（API 请求 300次/15分钟，登录尝试 20次/小时）需通过压力测试脚本进行验证，确保其在面临恶意请求时能有效拦截。

部署与灾备预案是保障系统持续运行的基石。当前的部署脚本中已包含数据库备份命令，上线前必须在服务器上配置 Linux crontab，实现 SQLite 数据库文件的每日定时备份，并自动上传至阿里云 OSS。由于系统采用 Docker 容器化部署，运维团队需保留上一个稳定版本的镜像标签，以便在发现致命缺陷时能够迅速修改 `docker-compose.prod.yml` 文件并完成版本回滚。

## 4. 测试工程师视角：边界测试与自动化建议

对于当前规模的系统，虽然全量的 UI 自动化测试维护成本过高，但针对核心计算逻辑和关键业务路径，必须依赖精心设计的测试用例和轻量级的自动化脚本来提供保障。

### 4.1 核心测试用例清单

以下表格列出了系统上线前必须通过的核心测试用例，涵盖了权限、薪酬、钱包和休假等关键模块。

| 测试模块 | 测试场景描述 | 预期系统行为 | 优先级 |
| :--- | :--- | :--- | :--- |
| 权限隔离 | 管理员账号尝试通过 API 直接访问 Client Portal 的数据端点 | 系统返回 401 或 403 错误，拒绝访问 | P0 |
| 数据隔离 | 客户 A 登录后，尝试通过修改 URL 参数查看客户 B 的员工详情 | 系统返回 404 或 403 错误，严格执行数据隔离 | P0 |
| 薪酬计算 | 员工于月中（如18号）入职，基础薪资为 3000 美元，当月共有 20 个工作日，实际工作 10 天 | 系统自动生成金额为 1500 美元的 Sign-on Bonus，并将其状态设为已批准，顺延并入次月薪资单 | P0 |
| 遗弃数据 | 已终止（terminated）状态的员工，其上月提交的报销在本月被管理员审批通过 | 定时任务自动锁定该报销，并触发兜底机制为该离职员工生成最终的结算薪资单 | P0 |
| 钱包扣款 | 客户钱包余额为 100 美元，系统生成了一张总额为 150 美元的发票并提交审核 | 钱包余额被扣减至 0，发票的待付金额（amountDue）更新为 50 美元 | P0 |
| 钱包回滚 | 将上述处于待审核状态的发票手动驳回至草稿状态 | 钱包自动退回 100 美元余额，发票的待付金额恢复为 150 美元 | P0 |
| 休假拆分 | 员工提交从 1 月 30 日至 2 月 2 日的跨月休假申请 | 系统自动将该申请拆分为两笔独立的记录：1.30-1.31 和 2.1-2.2，以适配按月计算的薪资逻辑 | P1 |
| 品牌呈现 | 触发密码重置或发送新员工邀请邮件 | 用户收到格式正确的 HTML 邮件，邮件中包含 GEA 官方品牌 Logo 且链接跳转正常 | P1 |

### 4.2 自动化测试脚本建议

为了在低维护成本的前提下获得最大的质量保障，建议测试团队重点开发以下两类 API 级别的集成与单元测试脚本：

首先是钱包并发测试脚本。测试工程师应编写一个基于 Node.js 的脚本，利用 `Promise.all` 构造 10 到 20 个并发请求，尝试同时扣减同一个客户钱包的余额。该脚本的验收标准是：在所有请求执行完毕后，钱包的最终余额必须绝对正确，且在并发请求中，只有符合余额条件的请求能够成功执行（或通过队列机制依次执行），其余因并发冲突或余额不足的请求必须返回明确的错误提示，绝不允许出现余额透支或数据相互覆盖的现象。

其次是薪资计算逻辑的单元测试。针对 `calculationService.ts` 和 `cronJobs.ts` 中的 Pro-rata（按比例）计算逻辑，建议补充完整的 Jest 测试用例。这些用例必须覆盖各种复杂的日历情况，包括但不限于大月（31天）、小月（30天）、二月（平年28天/闰年29天）以及入职或离职日期恰好落在周末或公共假期的特殊场景，确保系统在任何日历组合下都能得出精确到分（小数点后两位）的薪资计算结果。

## 5. 最终上线检查清单 (Go-Live Checklist)

在系统正式切换至生产环境并向客户开放之前，必须逐一确认以下所有事项均已完成并签字确认：

- 所有列出的 P0 和 P1 级别测试用例均已通过人工或自动化脚本验证，且无遗留的阻塞性缺陷。
- 生产环境配置文件（`.env.production`）中的各项敏感信息（包括 OSS 密钥、SMTP 凭证、JWT 密钥及正式域名）均已正确配置。
- 阿里云服务器的安全组/防火墙已正确开放 80 和 443 端口，且 Nginx 的 SSL 证书已成功申请并配置生效。
- 生产环境的 SQLite 数据库文件（`production.db`）已配置每日自动备份脚本，并验证了备份文件能够成功上传至安全的云存储中。
- 在预发环境（Staging）中，通过手动修改服务器系统时间，成功模拟执行过至少一次完整的“每月 5 号”定时任务流，确认薪资单和发票生成无误。
- 系统管理员的初始账号与高强度密码已妥善记录在安全的密码管理器中，并准备好在系统上线后安全地交付给客户。
