# GEA EOR SaaS 上线前测试与验收方案 (V2 完整版)

## 1. 验收策略概览

在初版方案的基础上，经过对代码库（包含 CRM 销售模块、AOR 承包商模块、通知中心、文档存储及安全配置等）的全面深度复核，本方案进一步完善了测试与验收的覆盖面。针对 GEA EOR SaaS Admin 系统高财务合规要求、多门户隔离以及复杂的定时流转机制，本方案从产品经理、开发工程师和测试工程师三个视角，提供了一套严密、可执行的上线前保障策略。

本方案不仅涵盖了核心的 EOR 员工生命周期，还补充了 CRM 销售转化、AOR 承包商计费、多渠道通知、文件存储安全以及前端体验等关键领域的验收标准。

---

## 2. 产品经理视角：全链路业务闭环与体验验收

产品经理的验收核心在于验证从“潜在客户”到“最终结算”的端到端业务流是否顺畅，以及多门户之间的交互体验是否符合预期。

### 2.1 CRM 销售与客户转化链路

业务的起点始于销售模块。验收时需验证销售线索（Sales Lead）在不同管道阶段（如 `discovery` -> `quotation_sent` -> `msa_signed` -> `closed_won`）的顺畅流转。特别是当线索转化为客户时，系统必须能够根据生成的报价单（Quotation）自动创建客户档案，并准确继承报价单中的服务费率（Service Fee）和一次性费用（One-time Fee）。此外，需验证报价单中的国家指南（Country Guide）附件是否能正确随邮件发送给潜在客户。

### 2.2 AOR 承包商与 EOR 员工双轨生命周期

系统支持 EOR（名义雇主）和 AOR（记录雇主/承包商）双轨业务。对于 EOR 员工，除了初版中提到的月中入职按比例计算 Sign-on Bonus、跨月休假拆分以及离职兜底机制外，还需重点验证“自助入职（Self-Onboarding）”流程。员工通过邀请链接进入无需登录的表单页面，填写的个人信息、银行账户及上传的合规文档必须能准确同步至 Admin 后台。

对于 AOR 承包商，验收重点在于不同计费模式的准确性。需分别验证按月（Monthly）、半月（Semi-monthly）和按里程碑（Milestone）计费的承包商，在到达计费周期或里程碑被标记为“已锁定（locked）”时，系统能否正确生成相应的承包商发票（Contractor Invoice），并最终汇总至客户的主发票中。

### 2.3 账单、钱包与财务分摊

账单与钱包是系统的财务核心。在验证发票状态流转（草稿 -> 待审核 -> 已发送 -> 已支付）和客户钱包自动扣款/回滚的基础上，需补充验证“冻结钱包（Frozen Wallet）”机制。当员工入职产生押金发票（Deposit Invoice）并支付后，资金应进入冻结钱包；当员工离职时，系统生成的信用票据（Credit Note）必须能准确将资金从冻结钱包释放回可用钱包。

此外，账单分摊（Allocation）模块也需严格验收。系统必须能够将供应商账单（Vendor Bills）精确分摊至对应的客户发票和具体员工身上，以确保利润与损失报表（Profit & Loss Report）中毛利计算的绝对准确。

### 2.4 多渠道通知与多语言体验

系统内置了基于模板的通知中心（Notification Service）。验收时需触发各类业务事件（如发票发送、休假审批），验证系统能否根据配置（`DEFAULT_RULES`）准确向 Client、Worker 或 Admin 发送对应的邮件和站内信（In-App Message）。邮件内容必须包含正确的动态变量（如发票金额、到期日）、多语言（中英双语）支持，并正确渲染 GEA 的品牌 Logo 和 PDF 附件。特别是对于包含中文字符的 PDF 发票，需验证其能否通过动态下载的 CJK 字体正确显示，避免出现乱码。

## 3. 开发工程师视角：架构安全、并发与健壮性验收

开发工程师的验收重点在于确保系统在复杂生产环境下的并发安全性、定时任务的准确调度、文件存储的可靠性以及整体架构的防护能力。

### 3.1 核心定时任务与时区一致性

系统的自动化高度依赖定时任务。除了验证任务的错峰执行（如 00:00 自动锁定，00:01 承包商发票生成，00:05 EOR 薪资单生成）外，必须确保整个 Node.js 进程和数据库操作严格遵循北京时间（UTC+8）。开发人员需验证 `cutoff.ts` 中的截止日期计算逻辑，确保“每月 4 号 23:59”的判定在跨月、跨年以及不同服务器系统时区下均表现一致，绝不容许出现因时区偏差导致的数据提前锁定或延迟生成。

### 3.2 并发控制与数据库事务

针对 Wallet Service（钱包服务）和 Frozen Wallet（冻结钱包），系统采用了基于 `version` 字段的乐观锁机制。上线前，必须通过编写高并发的 Node.js 测试脚本，模拟多个进程同时对同一客户的钱包进行充值、扣款或冻结操作。验收标准是：系统必须正确抛出并发冲突异常（并进行重试或拒绝），且在任何情况下都不能出现余额透支、数据覆盖或资金丢失。同时，所有涉及资金变动的操作必须被严格包裹在 `db.transaction` 中，确保业务逻辑与资金流水的原子性。

### 3.3 存储安全与访问控制

系统集成了阿里云 OSS（兼容 S3 API）进行文件存储。开发人员需验证 `storage.ts` 中的配置，确保敏感文件（如员工护照、合同、银行信息）被上传至私有 Bucket，并且前端只能通过临时签名的 URL（Signed URL）进行访问。同时，需检查文件上传接口，确保对文件大小（如限制为 10MB）和文件类型（MIME Type）进行了严格校验，防止恶意文件上传。

### 3.4 网络安全与防御机制

在应用层安全方面，需全面验证系统的防御机制。首先，确认 Nginx 配置中已启用 HTTPS，并且 `enable-ssl.sh` 脚本正确配置了 HSTS（Strict-Transport-Security）头部。其次，检查 Express 应用中 `helmet` 中间件的 CSP（内容安全策略）配置，确保有效防范 XSS 攻击。对于跨域认证，系统在三大子域名（admin, app, worker）间使用了 `sameSite: "none"` 和 `secure: true` 的 Cookie 配置，开发人员需在真实浏览器中验证跨域 JWT Token 的传递是否稳定。此外，管理员的 JWT Token 默认有效期为一年（365天），而 Portal/Worker 的有效期为 7 天，需确认此配置符合业务安全预期。

## 4. 测试工程师视角：边界测试与自动化保障

测试工程师需通过严密的用例设计和轻量级自动化脚本，覆盖系统中的边界条件和高风险路径。

### 4.1 核心边界测试用例清单

以下表格列出了系统上线前必须通过的 P0/P1 级别核心测试用例，覆盖了新增复核出的风险点。

| 测试模块 | 测试场景描述 | 预期系统行为 | 优先级 |
| :--- | :--- | :--- | :--- |
| 权限与越权 | Worker（员工）尝试通过修改 API 路径访问 Admin 或 Client Portal 的数据 | 系统返回 401/403 错误，严格拦截跨角色访问 | P0 |
| 并发资金 | 使用脚本发起 20 个并发请求，尝试使用仅有 100 美元余额的钱包支付多张发票 | 仅 1 个请求成功扣款，其余请求返回余额不足或并发冲突错误，余额不为负数 | P0 |
| AOR 计费 | 半月薪（Semi-monthly）承包商在当月 16 号触发定时任务 | 系统自动生成包含 1 号至 15 号基础费率及该期间已锁定异动的发票 | P0 |
| 押金流转 | 员工离职（状态转为 terminated），系统生成押金退还的 Credit Note 并审批通过 | 该员工对应的押金金额从客户的“冻结钱包”中扣除，并等额增加至“可用钱包”中 | P0 |
| 跨月休假 | 员工申请 2 月 28 日至 3 月 2 日的休假（平年） | 系统准确将休假拆分为 2.28（归属 2 月）和 3.1-3.2（归属 3 月）两条记录 | P1 |
| 文件安全 | 尝试上传一个伪装成 PDF 的可执行脚本文件（.sh 或 .exe）至合规文档模块 | 后端验证文件 MIME Type 失败，拒绝上传并返回明确错误 | P1 |
| PDF 渲染 | 生成包含中文姓名、中文职位及特殊符号的 Invoice PDF | 系统成功从 CDN 缓存并加载 CJK 字体，PDF 渲染清晰无乱码 | P1 |

### 4.2 自动化测试脚本建议

鉴于系统的核心价值在于财务与合规的准确性，建议测试团队优先开发以下自动化集成测试脚本，并将其集成至 CI/CD 流程中：

1. **钱包并发与事务测试脚本**：编写基于 Node.js 的高并发脚本，针对 `walletService.ts` 的 `transact` 和 `frozenWallet` 方法进行压力测试，确保乐观锁在高并发下的绝对可靠性。
2. **薪资与截止日期（Cutoff）单元测试**：针对 `cutoff.ts` 和 `calculationService.ts`，补充覆盖各种边界日期的 Jest 单元测试。例如：闰年的 2 月 29 日、跨年薪资计算、以及当服务器处于非 UTC+8 时区时，截止日期判定是否依然准确。
3. **发票金额汇总测试**：编写脚本模拟生成包含多个员工（含不同货币、Sign-on Bonus、按比例扣减休假）和多个 AOR 承包商的混合发票，验证主发票的总金额（Total Amount）是否严格等于所有子项金额之和，且汇率转换（USD -> XXX）精度无误。

## 5. 最终上线检查清单 (Go-Live Checklist)

在系统正式向客户开放之前，请项目负责人逐一核对并确认以下事项：

- [ ] 所有 P0 和 P1 级别测试用例（包含 EOR/AOR 双轨、钱包并发、权限隔离）均已通过验证，无遗留阻塞缺陷。
- [ ] 生产环境 `.env.production` 中的 JWT 密钥已重置，且阿里云 OSS（私有 Bucket）及 SMTP 凭证配置无误。
- [ ] 阿里云服务器防火墙已配置，`enable-ssl.sh` 已执行，三个子域名（admin, app, worker）的 HTTPS 及 HSTS 均已生效。
- [ ] 生产环境的 SQLite 数据库（`production.db`）已配置每日自动备份脚本，并成功测试了向 OSS 的备份上传与恢复流程。
- [ ] 在预发环境中，通过修改系统时间成功模拟了“每月 5 号 00:05”的定时任务，确认了 EOR 薪资单和 AOR 发票的自动生成逻辑。
- [ ] 管理员初始高强度密码已生成并安全保存；确认 Admin JWT 的 1 年有效期符合管理团队的安全策略。
