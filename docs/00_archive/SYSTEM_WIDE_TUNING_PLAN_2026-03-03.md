# 系统级调优方案（2026-03-03）

> 目标：在 Portal / Admin / API / 数据 / 运维全链路上，建立“可扩展、可观测、可运营、可恢复”的系统能力，而不只局部优化单一模块。

## 1. 产品与流程层（业务闭环）

### 1.1 全流程漏斗治理
- 为核心流程建立统一漏斗：
  - Portal 注册/登录 → Onboarding → 首次关键动作（如提交资料/查看发票/提交请假）
  - Knowledge 检索 → 点击 → 反馈 → 是否转人工
- 每个漏斗节点定义 `进入率/完成率/平均耗时/放弃率`。

### 1.2 用户分层体验策略
- 客户分层：新客户、稳定客户、高活跃客户。
- 角色分层：Admin、Customer Manager、Portal Contact。
- 对应提供不同默认首页与提醒密度，降低信息噪音。

### 1.3 通知策略统一
- 将系统内消息、邮件、Portal 提醒统一为策略中心：
  - 事件触发
  - 频率控制
  - 去重规则
  - 多语言模板

## 2. 应用层（前后端功能性与易用性）

### 2.1 表单与状态机一致性
- 所有关键业务对象（Invoice/Payroll/Leave/Adjustment/Onboarding）统一状态展示组件：
  - 当前状态
  - 可执行动作
  - 禁止原因
  - 审计记录入口
- 前端与后端状态机定义共享常量，避免“前端能点、后端拒绝”的断层。

### 2.2 统一错误语义
- 错误分级：
  - `UserActionable`（用户可修复）
  - `Retryable`（可重试）
  - `SystemFault`（系统异常）
- 统一用户提示文案与日志结构，减少“未知错误”。

### 2.3 移动端优先改造清单
- 针对高频移动流程（Portal Onboarding / Knowledge / Reimbursement）做专项适配：
  - 操作区可达性
  - 小屏密度
  - 异常提示可读性
  - 上传流程鲁棒性

## 3. 数据与AI层（质量、成本、可靠性）

### 3.1 数据质量守门（DQ Rules）
- 对关键表建立自动巡检：
  - 空值异常、状态非法跳转、金额负值、跨客户脏关联
- 产出每日质量报告并告警。

### 3.2 AI治理三件套
- 路由策略（Task Policy）
- 调用观测（Telemetry）
- 成本治理（Budget + 降级策略）
- 形成每周 AI 健康报告（成功率、回退率、P95、成本）。

### 3.3 Knowledge 生命周期
- 文章增加 `reviewAt/expiryAt`，支持自动复审和过期降权。
- 将内容缺口反馈纳入选题优先级系统。

## 4. 平台与运维层（SRE 基线）

### 4.1 可观测性三层
- 指标（Metrics）：请求量、错误率、P95、队列长度、任务失败率。
- 日志（Logs）：结构化日志，支持 traceId / customerId / userId / taskType。
- 链路（Trace）：关键 tRPC 与 AI 调用链打通。

### 4.2 稳定性工程
- 对关键外部依赖（LLM、汇率、S3）加：
  - 超时
  - 重试
  - 熔断
  - 降级
- 定义故障演练 Playbook（每季度至少一次）。

### 4.3 发布与回滚
- 建立分级发布策略：
  - 内部灰度
  - 小客户灰度
  - 全量
- 每次发布绑定回滚开关与数据库兼容策略。

## 5. 安全与合规层

### 5.1 访问与权限审计
- 每周自动扫描 RBAC 异常授权。
- 高风险操作（删除、发布、审批）强审计并可追踪操作者。

### 5.2 敏感信息治理
- 日志脱敏（邮箱、手机号、证件号）。
- AI prompt 输入输出敏感字段屏蔽。

### 5.3 供应链安全
- 依赖版本扫描、漏洞告警、补丁 SLA。

## 6. 工程效率层

### 6.1 质量门禁
- PR 必需通过：
  - Typecheck
  - Lint
  - 关键路径回归测试
- 失败阻断合并。

### 6.2 测试分层
- 单元测试：规则与计算
- 集成测试：tRPC + DB
- E2E：Portal 核心流程
- 引入测试稳定性指标（flaky rate）。

### 6.3 i18n 与可访问性持续治理
- 新增页面默认必须提供 EN/ZH 文案。
- 每月一次硬编码扫描与可访问性审计。

## 7. 推荐执行路线图（90天）

### 第1阶段（0-30天）
- 完成关键流程漏斗埋点
- 建立 AI telemetry 与健康看板
- 完成 Portal 高优先移动端页面修复

### 第2阶段（31-60天）
- 上线数据质量巡检与异常告警
- 建立统一错误语义与前后端状态机对齐
- 引入灰度发布与标准回滚模板

### 第3阶段（61-90天）
- 完成 Knowledge 生命周期治理
- 建立 SRE 仪表盘 + 故障演练机制
- 完成安全与依赖治理自动化

## 8. 成功标准（建议量化目标）

- Portal 核心流程完成率提升 ≥ 15%
- Knowledge 搜索无结果率下降 ≥ 20%
- AI 调用失败定位时长下降 ≥ 50%
- P95 API 响应下降 ≥ 20%
- 高优先级线上事故数下降 ≥ 30%
