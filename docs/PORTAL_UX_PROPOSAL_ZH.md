# 客户管理与 Portal 权限体验优化方案

本文档针对您提出的两个问题（问题 2：联系人 Role 与 Portal 权限割裂；问题 3：Access Client Portal 逻辑不清晰）进行深入分析，并结合两条硬规则提出完整的重构方案。

## 问题分析与当前现状

### 问题 2：客户联系人与 Portal 权限的割裂
目前在 Admin 的客户详情页中，联系人（Contacts）的配置存在严重的体验问题。首先，新建联系人时选择的 `Role` 实际上代表的是**业务职位**（如 HR Manager、Finance Manager、CEO 等），但这很容易让管理员误以为是在配置 Client Portal 的系统权限。其次，真正的 Portal 权限（`portalRole`：admin, hr_manager, finance, viewer）**只能在生成邀请链接（Invite）的那一刻进行选择**。一旦邀请发送，管理员就无法在界面上修改该用户的 Portal 权限，除非撤销权限并重新发送邀请。此外，目前的联系人列表不支持内联编辑，管理员无法修改已创建联系人的姓名、邮箱或职位。

### 问题 3：Access Client Portal（模拟登录）逻辑的不确定性
目前 Admin 端的 "Access Client Portal" 按钮实现逻辑是：接收一个 `customerId`，然后在后端查询该客户下的所有联系人，**挑选第一个**具有 `isPortalActive` 和 `hasPortalAccess` 的联系人生成 JWT Token 进行模拟登录。这种设计的缺陷在于，如果一个客户有多个 Portal 用户，系统选择的联系人是随机的（非确定性）。管理员在点击按钮前，完全不知道自己将以哪个用户的身份进入 Portal。虽然操作成功后会弹出提示框告知登录身份，但这是一种“盲盒”体验，不符合企业级应用的安全与可控要求。

---

## 优化方案建议

为了解决上述问题，我们需要对数据结构和前端交互进行梳理，确保业务逻辑与系统权限彻底解耦，同时提升操作的确定性。

### 方案 1：解耦业务职位与系统权限，支持权限编辑

我们建议将“业务职位”与“系统权限”在 UI 上进行明确区分，并支持对已激活用户的权限进行动态修改。

| 优化点 | 具体实施方案 | 预期效果 |
|--------|--------------|----------|
| **字段重命名** | 在前端表单和表格中，将原有的 `Role` 标签修改为 `Job Title`（职位），并在下拉菜单中保留 CEO、HR Manager 等选项。 | 消除歧义，明确该字段仅代表客户方的行政职务，不涉及系统控制。 |
| **支持联系人编辑** | 为每个联系人行添加一个 `Edit` 按钮，弹出一个包含姓名、邮箱、电话和 `Job Title` 的编辑对话框。 | 解决当前无法修改联系人基本信息的痛点。 |
| **独立管理 Portal 权限** | 在联系人表格中新增 `Portal Role` 列（显示 admin, viewer 等）。对于已开通 Portal 权限的用户，提供一个独立的 `Change Permission` 操作。 | 管理员可以随时调整用户的 Portal 权限，而不需要繁琐的“撤销+重发邀请”流程。 |

通过上述调整，联系人的生命周期变得清晰：创建联系人（填写基本信息和职务） → 邀请进入 Portal（分配初始系统权限） → 后期维护（独立修改基本信息或系统权限）。

### 方案 2：确定性的 Access Client Portal 体验

针对模拟登录的盲盒问题，我们建议改变“由系统随机选择”的逻辑，将控制权交还给管理员。

| 优化点 | 具体实施方案 | 预期效果 |
|--------|--------------|----------|
| **指定主联系人登录** | 废弃当前的随机选择逻辑。当管理员点击全局的 `Access Client Portal` 按钮时，系统应**强制寻找被标记为 `isPrimary`（主联系人）且拥有 Portal 权限的用户**。如果主联系人没有权限，则阻止操作并提示“请先为主联系人开通 Portal 权限”。 | 确保全局入口的确定性，通常主联系人（Primary Contact）也是客户方的最高权限代表。 |
| **精确的行级模拟登录** | 在 Contacts 选项卡的表格中，为每一个已经激活 Portal 权限的联系人行，添加一个独立的 `Login As`（以此身份登录）按钮。 | 管理员可以精确选择以 HR、财务或普通员工的视角进入 Portal，方便排查特定权限下的视图问题。 |

这种设计不仅解决了不确定性，还为管理员提供了强大的排障工具——他们可以随时切换不同角色的视角来验证 Portal 的显示效果。

## 实施步骤与技术评估

实施上述方案**不需要修改数据库结构（No DB Migration）**，现有的 `customerContacts` 表已经包含了所需的所有字段（`role`、`portalRole`、`isPrimary`、`isPortalActive` 等）。所有的优化都集中在前后端接口的完善和 UI 组件的重构上。

1. **后端接口扩展**：
   - 完善 `customers.contacts.update` 接口，允许更新 `portalRole`。
   - 修改 `customers.generatePortalToken` 接口，支持传入可选的 `contactId` 以实现精确登录；如果不传，则强制寻找 `isPrimary === true` 的联系人。

2. **前端 UI 重构**：
   - 在 `Customers.tsx` 的 Contacts 选项卡中，重命名表头和表单标签。
   - 实现联系人的编辑对话框（复用现有的表单组件）。
   - 在表格操作列（Actions）中，为已激活用户添加 `Change Permission` 和 `Login As` 按钮。
   - 调整全局 `Access Client Portal` 按钮的点击事件，处理找不到主联系人时的错误提示。

如果您认可上述分析与方案，我们可以将这些优化作为一个独立的任务（Task）进行排期开发。这将极大提升管理员在配置客户系统体验时的效率和直观性。
